# 2026-07-16 本地模型启动与 CosyVoice 依赖检查更新

## 更新背景

本次更新主要改善 local 模式下的 CosyVoice 3.0 启动和故障诊断。此前项目可能使用只安装了 FastAPI 基础依赖的 `.venv` 启动，或者在缺少 `third_party\Matcha-TTS` 时直到导入模型阶段才报出不够明确的错误。

## 主要变化

### 1. 本地启动脚本优先使用 CosyVoice 环境

`run-local.ps1` 现在按以下顺序选择 Python：

1. `D:\shenggui\CosyVoice\cosy_env\Scripts\python.exe`
2. 项目目录下的 `.venv\Scripts\python.exe`

启动时会输出实际使用的 Python 路径，并提示 SenseVoice GGUF、CosyVoice 权重、运行依赖和 Matcha-TTS 都需要准备完整。

### 2. 健康检查增加 Matcha-TTS 状态

`GET /api/health` 的 `models.voice_clone` 现在额外返回：

- `matcha_tts`：预期的 Matcha-TTS 目录。
- `matcha_ready`：该目录下是否存在 `matcha` Python 包。

可以在启动模型前直接确认 CosyVoice 子模块是否完整。

### 3. CosyVoice 加载前主动校验 Matcha-TTS

模型适配器会在导入 CosyVoice 前检查：

```text
<CosyVoice repo>\third_party\Matcha-TTS\matcha
```

如果目录缺失，接口会返回明确错误，提示补齐 Matcha-TTS 内容，或者使用递归方式克隆 CosyVoice 仓库。

## 本地目录要求

默认配置期望以下结构：

```text
D:\shenggui\CosyVoice\
  cosy_env\
    Scripts\
      python.exe
  CosyVoice-main\
    third_party\
      Matcha-TTS\
        matcha\
  CosyVoice3-5B\
    cosyvoice3.yaml
    llm.pt
    flow.pt
    hift.pt
    speech_tokenizer_v3.onnx
```

SenseVoice llama 运行时仍需要：

```text
D:\shenggui\SenseVoicePublic\
  llama-funasr-sensevoice.exe
  sensevoice*.gguf
```

## 启动与验证

在项目目录执行：

```powershell
.\run-local.ps1
```

服务启动后检查：

```powershell
Invoke-RestMethod http://127.0.0.1:8000/api/health |
    ConvertTo-Json -Depth 6
```

重点确认：

- `backend` 为 `local`。
- `models.voice_clone.matcha_tts` 指向正确目录。
- `models.voice_clone.matcha_ready` 为 `true`。
- 启动终端输出的 Python 路径是预期的 CosyVoice 环境。

`matcha_ready=true` 只表示 Matcha-TTS 目录完整，不代表所有模型权重、Python 依赖或 CUDA 环境都已就绪。

## 兼容性

- mock 模式不受影响。
- 如果 CosyVoice 专用环境不存在，启动脚本仍会回退到项目 `.venv`。
- `SHENGGUI_MODEL_FALLBACK_TO_MOCK=true` 时，本地模型加载失败仍可回退到 mock 响应。
